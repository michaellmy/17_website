About Hospital Surveys

Introducing: Hospital Surveys


 
 

Our Motivations:


Hear the voices of these children.

We believe it is crucial for child patients of Great Ormond Street Hospital to be able to provide feedback based on their experiences. Whether it be hospital facilities, the way patients are treated, or just the overall experience, feedback not only allows children to share their experiences, but can also help improve hospital services in the future.


Hospital Surveys - Our Solution.
Thus, we have developed Hospital Surveys, an online survey tool that that allows child patients to answer surveys to provide feedback anonymously. We aspire to eliminate the need for filling paper forms manually, while providing child patients a safe space to express themselves.


 

 

Project Abstract:

Hospital Surveys is an online survey tool for hospitals. Surveys are designed by hospital staff using a web application and delivered to patients using a progressive web app. A database stores the anonymised patient feedback, which can be exported for further analysis.

Key Features:

  • Progressive Web Application.
  • App allows admins to design, create, and manage questionnaires stored in a database, in turn allowing child patients to answer these questionnaires.
  • Data analysis of questionnaire data and responses
  • Django back-end REST API, rendered with a React JS front-end User Interface.
  • Web app is hosted on Heroku.

Related Frameworks, Technologies, and Projects

Related Technologies


Since our application is a completely new web-application, it was important for us to do some research on the various frameworks and libraries to be used on our web-app. The frameworks that we have researched have their own advantages and disadvantages, but after spending more time learning about these frameworks, we have narrowed them down to a few choices that suited our needs:


  • Django

    Django is a Python framework which follows the model-template-view pattern. In practice, we found it to have many built-in features such as a back-end administration panel UI, easy database mapping, and easy authentication [1].

    Django's ORM (object-relational mapping) layer handles database migrations very elegently, using Models to create tables instead of requiring users to use SQL directly [6]. Since we are required to store survey data in a database, this feature could be very helpful.

    Another advantage of using Django is that it is very secure [6]. This web framework provides protections against XSS and CSRF attacks, SQL injections, clickjacking and many more.

    For its downsides, time is required to get started to learn the basic Django boilerplate and structure, and Django can also come across as a little bit monolithic.

  • Flask

    Flask is a light-weight Python microframework that is easier to get started with [7]. It is unopinionated, and allows the developers to implement things using their own methods. .

    One advantage of using Flask is that it allows for a good learning experience as it forces us to truly understand our code implementation due to its flexibility.

    Besides that, Flask also has great performance, due to it being a micro framework. It is more "low-level" than Django, and contains fewer layers of abstraction between the web application and the database and HTTP requests.

    However, it offers much less built-in functionality than Django, and requires developers to be flexible [1] to make the best design decisions. Functionalities such as authentication is much harder to be implemented in the future.

  • Vue

    • Open-source JavaScript progressive framework, using the Model–view–Viewmodel pattern.
    • Heavy emphasis on Single File Components, allowing us to build style and functions within one file.

      •  
    • Fast and light-weight.
    • Has a gentle learning curve [2] due to its simple API.

  • React

    • A popular framework, hence research and resource availability is high.
    • Requires the use of various other libraries for different functionalities.

      •  
    • Steeper learning curve [2] than Vue, but ‘create-react-app’ package helps us get started.
    • Contains service workers [3] that allows users to see results even when offline.

 

Back-End Conclusion:

We have decided to use Django over Flask despite its steeper initial learning curve. This is because we believe Django lets us focus on the actual questionnaire designs in the long run, as a lot of functionality such as authentication and database management is already built-in. We also think that Flask may introduce many bugs in the future due to our inexperience in back-end development so far.



Front-End Conclusion:

Both frameworks were reasonable to us, but we decided that React was the best choice as our front-end JavaScript framework in the end. Despite having a steeper learning curve, we decided that the ‘create-react-app’ package can help us overcome this problem. The service workers also prove to be an advantage for developing a progressive web app.




Related Projects


  • SurveyMonkey

    SurveyMonkey is a website designed particularly for admins to design questionnaires. It is written in Python and developed with the .NET framework. It offers a wide range of functionalities, including functionalities to import questionnaires from templates, analyzing results, and obtaining verified questionnaire responses.


    It is very similar to our project, Hospital Surveys, as it allows for questionnaires that are highly customizable[4]. However, not all of these functionalities will be implemented, as many of these features are outside the scope of our project.

  • Google Forms

    Google forms is also designed for users to design questionnaires, and for users to respond to these questionnaires easily[5] and efficiently. Google Forms allows for admins to design questionnaires in a short amount of time. Due to its popularity, users are able to obtain a large amount of responses easily.


    We plan to implement a similar, simple UI design for users to design questionnaires, but we require more detailed ways to process and store the data from the questionnaire responses, i.e. through a database.


Conclusion

Google Forms offers less customization and functionality than SurveyMonkey but is much more user-friendly. On the other hand, SurveyMonkey is more focused on the questionnaires itself, and places more emphasis on the users creating the questionnaires. They both have their own merits. Google Forms' straight-forward UI and SurveyMonkey's customizability were both great features to have for our web-app.

Development Dilemma

Research On a Difficult Decision

App Development: Two Approaches


We had two approaches in developing the web application:
 a) We could develop the web-app using a stand-alone Python library such as Django that can also handle template rendering.
 b) We could use separate frameworks for front-end and back-end development, with a Javascript front-end consuming a Python back-end REST API.


Thus, we decided to compare both approaches:

Using Stand-Alone Django (Or Other Frameworks):
+ Time saving:
Django provides templates for rendering views, so developers are not required to code the front-end manually.
+ Convenience:
Features such as form validation and rendering may be more convenient to implement without a front-end framework. [9]
Back-end REST API with JavaScript Front-End:
+ Decoupling / Separation of Concerns [8]:
Separate frameworks allows us to split our work easily and evenly between front-end and back-end developers, preventing interference [9], which is a great software engineering practice.
+ Dynamic / Interactive User Interface:
Many JavaScript front-end frameworks allow for more dynamic and interactive UIs, leading to the final web-app looking more elegant and eye-catching.
+ Web-App Performance:
A separate JavaScript front-end can render separately from the server, only reloading the page when necessary [8] (when it is required to fetch data). This reduces loading times of the web-app.

Conclusion

While it is convenient not to use a front-end framework, the team agreed that using one would allow for a better learning experience and an overall higher quality web-app. Thus, we decided to use separate frameworks for front-end and back-end development.

User Authentication

Django's rest-auth library was used to generate a token for the front-end UI when users sign in using Django Admin credentials. React Redux then stores this token for 60 minutes in the browser. This creates a session that lasts for 60 minutes for users that have signed in, and allows users to navigate to any web page requiring authentication during that time without signing in again.

Testing

Testing Web-App Functionalities

Testing Strategy

We used multiples strategies when testing our software:
  1. Testing Often:
    We made sure to always test our web-app frequently, to ensure that all functionalities are working normally at all times.
  2. Regression Testing:
    When adding new functionalities, we always made sure that these new funcionalities did not break any existing functionalities.
  3. Usage of Virtual Environments:
    Virtual Environments proved to be very helpful, as it allows us to test dependencies and useful packages without installing them within our devices, saving a lot of time and device memory.
  4. Leaving Time for Fixing:
    Before testing our functionalities, we also made sure that there was enough time for us to fix any broken code or functionalities after the testing process.


Responsive Web Design Testing

Since our web-app allows child patients to answer questionnaires from tablets and mobile devices, our UI designs had to be tested to be responsive. The UI design frameworks we used (React Bootstrap / Ant Design) were very helpful here as their page grid system and responsive components could be used.

Stress testing was used to test the performance of our web-app. There may be possibilities where admins need to manage large number of questionnaires, causing huge loads of data in the database.

Thus, we tested the time taken for SQL queries in database tables with respect to number of rows in a test server. The number of rows represents the size of questionnaire and survey data.

Algorithms

Statistical Analysis Algorithms

K-Means Clustering

For the statistical analysis part of our web-app, we decided to implement a basic and common unsupervised Machine Learning algorithm - K-Means Clustering. This algorithms is used to analyze the age of the survey respondents by classifying and grouping respondent ages based on clustering patterns. Each group represents an (clustered) estimated age group of survey respondents grouped by the algorithm.
 
 

Experiment Setup:

Since we were not provided with a specific dataset by our clients, we had to manually fill in the survey forms ourselves to obtain some questionnaire response data. This resulted in a rather small sample size, and may be one of the compromises of using this algorithm. We implemented this algorithm by using the npm library skmeans, which allowed us to calculate the clusters easily.

However, due to shortage of time, we were only able to use this algorithm for uni-dimensional data (age). This is because multi-dimensional data was difficult to incorporate with Chart.js, and introduced some time complexity problems at the same time.
 
 

Experiment Results:

When implementing this algorithm, we had to choose the number of clusters (groups), k, to classify the data into. This value is the Hyperparameter of this algorithm. Shown below are the results of different values of k, ranging from values 2-4:
 

 
 
 

Discussion / Conclusions:

The algorithm seems to perform the best when clustering our data into 2 clusters (younger and older patients), which may be the optimal number in our case. This may be due to our small sample size, as larger values of k may force the algorithm to cluster data with more uncertainty. Also, the algorithm fails when the sample size is lower than k (the number of clusters), which occurs when there are very fewe responses to the surveys.

Hence, in order to improve the performance of this algorithm or use larger values of k, the most direct solution is to provide a larger sample size.

Deployment Manual (Heroku):


Installation Pre-Requisites:
  1. Heroku Command Line Interpreter (CLI)
  2. Heroku Account
  3. Git

Deployment Guide:
  1. Download the code from this repository, by downloading the zip file, or running the command:

    2. $ git clone https://github.com/michaellmy/hospital-surveys-dev.git

  2. Create a new Heroku project (free), then navigate into the directory containing manage.py.

    3. $ cd hospital-surveys-dev

  3. Log into the Heroku CLI, then initialize a git repository in a new or existing directory. These few instructions can also be found in Heroku's project dashboard.

    4. $ heroku login

    $ git init

    $ heroku git:remote -a [heroku-project-name]

  4. Deploy the application.

    5. $ git add .

    $ git commit -m "Initial commit message"

    $ git push heroku master

  5. Create a new superuser, and enter the credentails when prompted. These credentials will be used to log in to the admin panel.

    6. $ heroku run python manage.py createsuperuser

    The web app can now be found at https://[heroku-project-name].herokuapp.com. A Heroku Postgres database will be automatically attached to the web-app during deployment, and the credentials for the database can be found under the Heroku Project -> Resources -> Add-ons -> Heroku Postgres -> Settings -> View Credentials.
